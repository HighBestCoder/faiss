# Benchmark 1: HNSW FP32 vs FP16 vs BF16

## 测试环境

- **CPU**: Intel Xeon Platinum 8272CL (Cascade Lake), 8 cores
- **RAM**: 32 GB
- **FAISS**: 1.13.2 (源码编译, AVX2)
- **数据集**: Cohere Large 10M, 768维, cosine metric
- **HNSW 参数**: M=32, efConstruction=64
- **OMP 线程**: 8
- **Ground Truth**: 子集测试使用 brute-force IndexFlatIP 在对应子集上重新计算

## 综合结论

### FP16 是最佳选择

1. **内存**: 比 FP32 节省 46%，每向量 1808 B (vs FP32 的 3344 B)
2. **Recall**: 损失极小，R@10 在所有测试规模和 efSearch 下均 < 0.5% (vs FP32)
3. **QPS**: 整体比 FP32 快 5-34%，得益于更小的数据量带来的 cache 友好性
4. **构建时间**: 与 FP32 基本持平
5. **可扩展性**: 1M → 3M 趋势一致，结论可推广到更大规模

### BF16 不推荐

1. **Recall 损失明显**: 比 FP16 差 0.3-2.1%，约为 FP16 (vs FP32) 损失的 3-5 倍
2. **无硬件加速**: Cascade Lake CPU 没有 AVX-512 BF16 指令，BF16→FP32 转换靠软件 bit-shift
3. **精度不足**: 7-bit 尾数对 [-1,1] 范围的 cosine similarity 区分度不够
4. **索引大小相同**: 和 FP16 一样都是 2 字节/维度，没有额外存储优势

### 现有 IndexHNSWSQ 已足够

FAISS 内置的 `IndexHNSWSQ(QT_fp16)` 通过 ScalarQuantizer 的 decode-compute 流水线，性能已经很好。我们编写的自定义 SIMD 距离函数 (`fp16vec_inner_product` 等) 可作为未来进一步优化的基础，但对当前场景非必需。

### 10M 规模预估

基于 1M 和 3M 数据的线性外推:
- **FP16 索引大小**: ~17.2 GB (10M × 1808 B/vec)
- **FP16 构建时间**: ~2000s (~33 分钟)
- **FP16 R@10 (ef=256)**: 预计 ~0.96+ (recall 随数据量增大轻微下降)
- **需要**: ≥ 24 GB RAM (向量 + 索引)，当前环境 (32 GB) 内存不足以同时容纳原始 float32 向量和 FP16 索引
