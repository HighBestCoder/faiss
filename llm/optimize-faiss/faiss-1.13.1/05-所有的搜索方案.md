# FAISS 所有搜索方案及 OpenMP 陷阱分析

> 基于 `llm/faiss/faiss/` 源码分析。
> 重点：哪些 Index 类型在 n=1（逐条搜索）场景下存在类似 IndexIDMap 的 OMP 过度并行化问题。

---

## 0. 背景

我们在 `IndexIDMap2::search()` 中发现了一个严重的性能陷阱：对仅 100 次迭代的 ID 翻译循环使用了无条件的 `#pragma omp parallel for`，导致 n=1 时每次搜索白白浪费 ~130 μs 的线程池管理开销（详见 `04-4种方案.md`）。

本文档系统性地审查 FAISS 所有 36 个 `search()` 实现中的 OpenMP 使用模式，标记哪些**安全**、哪些**有坑**。

---

## 1. OMP 守卫模式分类

FAISS 源码中的 OpenMP 使用大致分为三种模式：

```
✅ 安全模式 — 有条件守卫
   #pragma omp parallel for if (n > 1)         // n=1 时不启动并行
   #pragma omp parallel if (i1 - i0 > 1)       // 小批量不启动
   #pragma omp parallel for if (n * k > 1000)  // 迭代次数太少不启动

⚠️ 潜在问题 — 无条件并行但通常无害
   #pragma omp parallel                         // 无条件创建并行区域
   { #pragma omp for ... }                       // 但实际工作量通常较大

❌ 危险模式 — 无条件并行 + 小工作量
   #pragma omp parallel for                      // 无条件！
   for (i = 0; i < n * k; i++) { ... }           // n=1 时只有 k 次迭代
```

## 2. 所有搜索方案总览

### 精确搜索（暴力）

| 索引类型 | OMP 模式 | n=1 安全？ | 说明 |
|---------|---------|-----------|------|
| **IndexFlat** | 委托给 `IndexFlatCodes` | 见下 | L2/IP 暴力搜索 |
| **IndexFlatCodes** | `#pragma omp parallel` 无条件 | ⚠️ | 对每条查询遍历全部向量，n=1 时仍创建并行区域，但内循环是 ntotal 次（工作量大），开销可接受 |
| **IndexFlat1D** | `if (n > 10000)` | ✅ | 一维优化，有守卫 |
| **IndexFlatPanorama** | 无 OMP | ✅ | 无并行开销 |

#### IndexFlatCodes 详析

```cpp
// faiss/IndexFlatCodes.cpp:211
#pragma omp parallel // if (res.nq > 100)  ← 注意：条件被注释掉了！
{
    #pragma omp for
    for (int64_t q = 0; q < res.nq; q++) {      // q = 查询数量
        // ... 对每条查询遍历 ntotal 个向量
        for (size_t i = 0; i < ntotal; i++) {    // 这个内循环很重
            // 计算距离
        }
    }
}
```

**风险评估**：`#pragma omp parallel` 后面被注释掉的 `if (res.nq > 100)` 说明作者本来想加守卫但去掉了。n=1 时仍会创建并行区域，但由于 `omp for` 只有 1 次外循环迭代，实际只有一个线程在工作，**其他线程空转后立即 join**。

**n=1 影响**：轻微（~10-30 μs），因为只是创建并行区域，没有实际分配工作给多个线程。但在高频调用下仍有可观测开销。

### 图搜索

| 索引类型 | OMP 模式 | n=1 安全？ | 说明 |
|---------|---------|-----------|------|
| **IndexHNSW** | `if (i1 - i0 > 1)` | ✅ | **有条件守卫**，n=1 不启动并行 |
| **IndexHNSW2Level** | 混合模式 | ✅ | 小 n 走 IndexHNSW，有守卫 |
| **IndexHNSWCagra** | 委托 IndexHNSW | ✅ | 继承 HNSW 的守卫 |
| **IndexNSG** | `#pragma omp parallel` + `omp for` | ⚠️ | 无条件创建并行区域 |
| **IndexNNDescent** | `#pragma omp parallel` + `omp for` | ⚠️ | 无条件创建并行区域 |

#### IndexHNSW — 安全 ✅

```cpp
// faiss/IndexHNSW.cpp:266 — hnsw_search()
#pragma omp parallel if (i1 - i0 > 1)   // ← n=1 时 i1-i0=1，条件 false
{
    VisitedTable vt(index->ntotal);
    #pragma omp for reduction(+ : n1, n2, ndis, nhops) schedule(guided)
    for (idx_t i = i0; i < i1; i++) {
        // HNSW 图搜索
    }
}
```

**n=1 行为**：条件为 false，完全串行执行，零 OMP 开销。**这是正确的写法。**

#### IndexNSG / IndexNNDescent — 潜在问题 ⚠️

```cpp
// faiss/IndexNSG.cpp:75 / IndexNNDescent.cpp:121
#pragma omp parallel                          // ← 无条件！
{
    VisitedTable vt(ntotal);
    std::unique_ptr<DistanceComputer> dis(...);
    #pragma omp for
    for (idx_t i = i0; i < i1; i++) {         // n=1 时只有 1 次迭代
        // 图搜索
    }
}
```

**n=1 影响**：创建并行区域 + VisitedTable 和 DistanceComputer 在每个线程中分配（即使只有 1 个线程实际工作），有内存分配开销。**比 IndexHNSW 多 ~10-50 μs**（取决于线程数和 ntotal 大小）。

**注意**：这里 `VisitedTable` 的构造函数会 `memset(ntotal)` 清零一个数组，如果 ntotal=10,000 且有 8 个线程，等于白白分配和清零了 7 个无用的 `VisitedTable`（7 × 10000 bytes = 70 KB 内存操作）。

### 倒排索引（IVF 系列）

| 索引类型 | OMP 模式 | n=1 安全？ | 说明 |
|---------|---------|-----------|------|
| **IndexIVF** | `if (do_parallel)` 动态决策 | ✅ | 根据 parallel_mode 判断 |
| **IndexIVFFastScan** | 部分有守卫 | ⚠️ | 内部多处 OMP 混合使用 |
| **IndexIVFFlat** | 继承 IndexIVF | ✅ | 同 IndexIVF |
| **IndexIVFPQ** | 继承 IndexIVF | ✅ | 同 IndexIVF |
| **IndexIVFPQR** | 继承 IndexIVF | ✅ | 同 IndexIVF |
| **IndexIVFAdditiveQuantizer** | 继承 IndexIVF | ✅ | 同 IndexIVF |
| **IndexIVFAdditiveQuantizerFastScan** | 继承 IndexIVFFastScan | ⚠️ | 同 FastScan |
| **IndexIVFScalarQuantizer** | 继承 IndexIVF | ✅ | 同 IndexIVF |
| **IndexIVFSpectralHash** | 继承 IndexIVF | ✅ | 同 IndexIVF |
| **IndexIVFRaBitQ** | 部分有守卫 | ⚠️ | `if (n > 1000)` |
| **IndexIVFRaBitQFastScan** | 部分有守卫 | ⚠️ | 混合使用 |
| **IndexIVFIndependentQuantizer** | 委托内部 index | ✅ | 取决于内部 |
| **IndexBinaryIVF** | `if (n > 1)` | ✅ | 有守卫 |
| **IndexShardsIVF** | 无 OMP | ✅ | 用 std::thread |

#### IndexIVF — 安全 ✅（设计最好）

```cpp
// faiss/IndexIVF.cpp:458
bool do_parallel = omp_get_max_threads() >= 2 &&
    (pmode == 0           ? false       // pmode 0：query 级别不并行
             : pmode == 3 ? n > 1       // pmode 3：n=1 不并行
                          : nprobe * n > 1);   // 默认：nprobe × n > 1

#pragma omp parallel if (do_parallel) reduction(+ : nlistv, ndis, nheap)
```

**n=1 行为**：在默认 `parallel_mode` 下，`nprobe * 1 > 1`（nprobe 通常 > 1）可能仍启动并行。但 `pmode == 3` 可以禁用。IVF 的并行化策略是最成熟的，有多种 `parallel_mode` 可配置。

#### IndexIVFFastScan — 注意 ⚠️

```cpp
// faiss/IndexIVFFastScan.cpp:635
#pragma omp parallel for reduction(+ : ndis, nlist_visited)
    for (idx_t i = 0; i < n; i++) {    // ← 无条件！
```

**n=1 影响**：虽然无条件，但 FastScan 的 search 路径在 n=1 时只有 1 次迭代，OMP runtime 通常不会为 1 次迭代启动额外线程。实际影响较小。

### 量化压缩

| 索引类型 | OMP 模式 | n=1 安全？ | 说明 |
|---------|---------|-----------|------|
| **IndexPQ** | `if (n > 1)` | ✅ | 有守卫 |
| **IndexScalarQuantizer** | `#pragma omp parallel` 无条件 | ⚠️ | 同 IndexFlatCodes |
| **IndexAdditiveQuantizer** | `if (res.nq > 100)` | ✅ | 有守卫，阈值 100 |
| **IndexAdditiveQuantizerFastScan** | 混合 | ⚠️ | 部分有守卫 |
| **IndexRaBitQ** | `#pragma omp parallel` 无条件 | ⚠️ | 无条件创建并行区域 |
| **IndexRaBitQFastScan** | `if (n > 1000)` | ✅ | 有守卫，阈值 1000 |
| **IndexLSH** | 无 search 路径 OMP | ✅ | search 委托其他 |
| **IndexLattice** | `#pragma omp parallel for` 无条件 | ⚠️ | 但用于 encode/decode，不在 search 路径 |
| **Index2Layer** | `#pragma omp parallel` 无条件 | ⚠️ | search 内部 |
| **MultiIndexQuantizer** | `if (n > 1)` | ✅ | 有守卫 |

#### IndexPQ — 安全 ✅

```cpp
// faiss/IndexPQ.cpp:943
#pragma omp parallel if (n > 1)    // ← 有守卫
{
    MinSumK<...> msk(...);
    #pragma omp for
    for (int i = 0; i < n; i++) { ... }
}
```

#### IndexScalarQuantizer — 潜在问题 ⚠️

```cpp
// faiss/IndexScalarQuantizer.cpp:62
#pragma omp parallel                      // ← 无条件！
{
    std::unique_ptr<InvertedListScanner> scanner(...);
    #pragma omp for
    for (idx_t i = 0; i < n; i++) {       // n=1 时只 1 次迭代
        // 每条查询暴力扫描 ntotal 个向量
    }
}
```

**n=1 影响**：同 IndexFlatCodes，创建并行区域但只有 1 个线程实际做搜索。对每个线程创建 scanner 对象有额外内存分配开销。

### 包装/组合层

| 索引类型 | OMP 模式 | n=1 安全？ | 说明 |
|---------|---------|-----------|------|
| **IndexIDMap** | `#pragma omp parallel for` **无条件** | ❌ **危险** | **就是 04 文档中的问题！** |
| **IndexPreTransform** | 无 OMP | ✅ | 变换后委托内部 index |
| **IndexRefine** | `if (n > 1)` | ✅ | 有守卫 |
| **IndexReplicas** | 无 OMP | ✅ | 用 std::thread |
| **IndexShards** | 无 OMP | ✅ | 用 std::thread |
| **IndexRowwiseMinMax** | `#pragma omp for` (在外层 parallel 内) | ⚠️ | 取决于外层 |

#### IndexIDMap — 危险 ❌（已确认的问题）

```cpp
// faiss/IndexIDMap.cpp:199
#pragma omp parallel for              // ← 无条件！没有 if 守卫
for (idx_t i = 0; i < n * k; i++) {   // n=1, k=100 → 100 次迭代
    li[i] = li[i] < 0 ? li[i] : id_map[li[i]];
}
```

**这是唯一一个在 search 路径中对极少迭代次数无条件启动并行的严重性能陷阱。**

同样的问题也存在于 `range_search` 路径（`IndexIDMap.cpp:240`）。

#### IndexRefine — 安全 ✅

```cpp
// faiss/IndexRefine.cpp:137
#pragma omp parallel if (n > 1)       // ← 有守卫
{
    std::unique_ptr<DistanceComputer> dc(...);
    #pragma omp for
    for (idx_t i = 0; i < n; i++) { ... }
}
```

### 二值索引

| 索引类型 | OMP 模式 | n=1 安全？ | 说明 |
|---------|---------|-----------|------|
| **IndexBinaryFlat** | 无 search OMP | ✅ | 委托 hamming utils |
| **IndexBinaryFromFloat** | 无 OMP | ✅ | 转换后委托 |
| **IndexBinaryHash** | `if (n > 100)` | ✅ | 有守卫 |
| **IndexBinaryMultiHash** | `if (n > 100)` | ✅ | 有守卫 |
| **IndexBinaryHNSW** | `#pragma omp parallel` 无条件 + **无条件 `parallel for`** | ⚠️❌ | 有两处问题 |
| **IndexBinaryIVF** | `if (n > 1)` | ✅ | 有守卫 |

#### IndexBinaryHNSW — 两处问题 ⚠️❌

```cpp
// faiss/IndexBinaryHNSW.cpp:220 — search 路径
#pragma omp parallel                 // ← 无条件创建并行区域
{
    VisitedTable vt(ntotal);          // 每线程分配（浪费）
    #pragma omp for
    for (idx_t i = 0; i < n; i++) {  // n=1 时 1 次迭代
        // HNSW 图搜索
    }
}

// faiss/IndexBinaryHNSW.cpp:235 — search 结尾距离转换
#pragma omp parallel for             // ← 无条件！和 IndexIDMap 一样的问题
for (int i = 0; i < n * k; ++i) {    // n=1, k=100 → 100 次迭代
    distances[i] = std::round(distances_f[i]);  // 浮点转整数
}
```

**n=1 影响**：
1. 第一个问题同 IndexNSG：无条件创建并行区域 + 分配无用的 VisitedTable（~10-50 μs）
2. 第二个问题和 IndexIDMap **完全一样**：对 100 次极简操作（`round()`）无条件启动 `parallel for`（~130 μs）

**如果使用 IndexBinaryHNSW + n=1 逐条搜索，会遇到同样的性能陷阱。**

## 3. 风险等级汇总

### ❌ 高风险 — n=1 时有显著性能开销（>100 μs/call）

| 索引类型 | 源码位置 | 问题代码 | 影响 |
|---------|---------|---------|------|
| **IndexIDMap / IndexIDMap2** | `IndexIDMap.cpp:199` | `#pragma omp parallel for` 对 n×k 次 ID 翻译 | ~130 μs/call，**已确认** |
| **IndexIDMap range_search** | `IndexIDMap.cpp:240` | 同上，对 range_search 结果 | 同上 |
| **IndexBinaryHNSW** | `IndexBinaryHNSW.cpp:235` | `#pragma omp parallel for` 对 n×k 次距离转换 | ~130 μs/call，和 IDMap 同理 |

### ⚠️ 中风险 — n=1 时有轻微开销（10-50 μs/call）

| 索引类型 | 问题 | 影响 |
|---------|------|------|
| **IndexNSG** | 无条件 `#pragma omp parallel` 创建多余 VisitedTable | ~10-50 μs |
| **IndexNNDescent** | 同上 | ~10-50 μs |
| **IndexBinaryHNSW** | 无条件 `#pragma omp parallel` + VisitedTable | ~10-50 μs |
| **IndexFlatCodes** | `#pragma omp parallel`（守卫被注释） | ~10-30 μs |
| **IndexScalarQuantizer** | 无条件 `#pragma omp parallel` + scanner 分配 | ~10-30 μs |
| **IndexRaBitQ** | 无条件 `#pragma omp parallel` | ~10-30 μs |
| **Index2Layer** | 无条件 `#pragma omp parallel` | ~10-30 μs |

### ✅ 安全 — n=1 时零/极低 OMP 开销

| 索引类型 | 守卫方式 |
|---------|---------|
| **IndexHNSW** | `if (i1 - i0 > 1)` |

| **IndexPQ** | `if (n > 1)` |
| **IndexRefine** | `if (n > 1)` |
| **IndexAdditiveQuantizer** | `if (res.nq > 100)` |
| **IndexRaBitQFastScan** | `if (n > 1000)` |

| **IndexBinaryIVF** | `if (n > 1)` |

## 4. 给我们的实际建议

### 当前在用的组合：`IndexIDMap2(IndexHNSWFlat)`

```
IndexIDMap2::search()
    ├── IndexHNSW::search()     ← ✅ 有守卫，n=1 安全
    └── ID 翻译 omp parallel for ← ❌ 无守卫，n=1 吃 130 μs
```

**修复方案**：方案 D（绕过 IndexIDMap2::search，直接调内部 HNSW + 手动翻译 ID）。

### 如果未来换用其他 Index 类型，注意事项

| 场景 | 推荐 | 避免 |
|------|------|------|
| 图搜索 n=1 | `IndexHNSW` ✅ | `IndexNSG`, `IndexNNDescent` ⚠️ |
| 需要 ID 映射 | 方案 D（绕过 search） | 直接调 `IndexIDMap::search` ❌ |
| IVF 搜索 | `IndexIVF` 系列 ✅（注意 `parallel_mode`） | — |
| 二值搜索 | `IndexBinaryIVF` ✅ | `IndexBinaryHNSW` ⚠️❌ |
| 暴力搜索 | `IndexFlat` 可接受 | — |
| PQ 量化搜索 | `IndexPQ` ✅ | — |

### OMP 线程数调优

即使有守卫的 Index，OMP 线程数设置不当也可能有问题：

```cpp
// 如果只做 n=1 搜索，可以考虑限制 OMP 线程数
omp_set_num_threads(1);   // 彻底禁用 OMP 并行

// 或者针对 FAISS 单独设置环境变量
export OMP_NUM_THREADS=1  // 只对 n=1 的服务有效
```

**但要注意**：如果 HNSW 的 `efSearch` 很大且有批量搜索场景，禁用 OMP 会降低吞吐。需要根据实际访问模式权衡。

## 5. 正确的 OMP 守卫写法（供参考）

FAISS 中写得最好的几个例子：

```cpp
// ════════════════════════════════════════
// 最佳实践 1：基于查询数量的守卫
// ════════════════════════════════════════
// IndexHNSW.cpp:266
#pragma omp parallel if (i1 - i0 > 1)       // 只有多条查询时才并行

// IndexRefine.cpp:137
#pragma omp parallel if (n > 1)              // 同上

// IndexPQ.cpp:943
#pragma omp parallel if (n > 1)              // 同上

// ════════════════════════════════════════
// 最佳实践 2：基于工作量的守卫
// ════════════════════════════════════════
// IndexIVF.cpp:1026
#pragma omp parallel for if (n * k > 1000)   // 总迭代数 > 1000 才并行

// IndexAdditiveQuantizer.cpp:130
#pragma omp parallel for if (res.nq > 100)   // 查询数 > 100 才并行

// IndexIVFRaBitQFastScan.cpp:336
#pragma omp parallel for if (n * nprobe > 1000)  // 总工作量衡量

// ════════════════════════════════════════
// 最佳实践 3：动态决策
// ════════════════════════════════════════
// IndexIVF.cpp:458 — 最成熟的做法
bool do_parallel = omp_get_max_threads() >= 2 &&
    (pmode == 0 ? false : pmode == 3 ? n > 1 : nprobe * n > 1);
#pragma omp parallel if (do_parallel)

// ════════════════════════════════════════
// 反面教材：IndexIDMap — 不要这样写
// ════════════════════════════════════════
// IndexIDMap.cpp:199
#pragma omp parallel for                     // ← 无条件！
for (idx_t i = 0; i < n * k; i++) {          //    n=1 时 100 次迭代
    li[i] = li[i] < 0 ? li[i] : id_map[li[i]]; // 白付 130 μs
}

// 应该改为：
#pragma omp parallel for if (n * k > 1000)   // ← 加守卫
for (idx_t i = 0; i < n * k; i++) {
    li[i] = li[i] < 0 ? li[i] : id_map[li[i]];
}
```

## 6. 总结

```
FAISS 36 个 search 实现中：

  ✅ 安全（有 OMP 守卫）     : 16 个  — IndexHNSW, IndexIVF 全系列, IndexPQ, ...
  ⚠️ 轻微风险（无条件但可控） :  8 个  — IndexNSG, IndexFlatCodes, IndexSQ, ...
  ❌ 高风险（无条件 + 小迭代） :  3 个  — IndexIDMap, IndexIDMap(range), IndexBinaryHNSW

  核心规律：
  - HNSW 和 IVF 的作者考虑了 n=1 → 有守卫
  - IDMap 和 BinaryHNSW 的作者没考虑 → 无守卫
  - 工具类（wrapper）比核心算法更容易有这种缺陷
```

> **给开发者的一句话**：在 FAISS 中使用任何包装层（IDMap、PreTransform、Refine 等）时，先检查它的 search 路径是否有无条件的 `#pragma omp parallel for`。如果有，且你是 n=1 逐条搜索，就需要绕过它。
