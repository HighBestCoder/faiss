## Memory Analysis

### Growth Pattern

- **Initial RSS**: 23.0 MB (empty server)
- **Final RSS (stable)**: 37,007.5 MB (~36.1 GB)
- **Total growth during loading**: 36,984.5 MB over 10M vectors
- **Average growth rate**: ~3,698 MB per million vectors (~3.61 GB/M vectors)
- **Bytes per vector**: ~3,882 bytes/vector (RSS-based)
  - Theoretical: 768 dims x 4 bytes = 3,072 bytes for raw float32 data
  - Overhead: ~810 bytes/vector (HNSW graph edges, id mapping, metadata)

### Memory Growth Rate Over Time

The memory growth rate is notably consistent through the entire loading phase, showing no significant deviations:

| Phase | Vectors | RSS Growth | Rate (MB/M vectors) |
|---|---|---|---|
| 0 - 1M | 0 → 1M | 23 → 5,022 MB | 4,999 MB/M |
| 1M - 2M | 1M → 2M | 5,022 → 8,826 MB | 3,804 MB/M |
| 2M - 3M | 2M → 3M | 8,826 → 12,063 MB | 3,237 MB/M |
| 3M - 5M | 3M → 5M | 12,063 → 19,129 MB | 3,533 MB/M |
| 5M - 7M | 5M → 7M | 19,129 → 25,618 MB | 3,245 MB/M |
| 7M - 9M | 7M → 9M | 25,618 → 32,338 MB | 3,360 MB/M |
| 9M - 10M | 9M → 10M | 32,338 → 35,803 MB | 3,465 MB/M |

The first million vectors show a higher per-vector cost (~5.0 GB) due to initial HNSW graph construction overhead and base allocations. After that, the rate stabilizes around 3.2-3.5 GB per million vectors.

### Memory Spike Detection

**No significant memory spikes detected.** The RSS grew monotonically throughout the loading phase with no sudden jumps or drops between consecutive 3-minute readings. The maximum inter-sample delta was ~760 MB (from 05:06:04 to 05:09:04, which is the initial burst of loading the first ~100K vectors + server initialization).

### FAISS std::vector Reallocation Analysis

Unlike some previous runs, **no 2x memory spike was observed** in this benchmark. This could indicate one of:

1. **Pre-allocated capacity**: The FAISS index may have been initialized with sufficient capacity to avoid reallocation during the 10M vector insertion.
2. **Incremental growth**: The `std::vector<uint8_t> codes` storage may have grown incrementally through multiple smaller reallocations that fell between the 3-minute memory sampling intervals.
3. **Memory sampling granularity**: With 3-minute sampling intervals, a transient spike (allocate 2x → copy → free old) could complete within seconds and be missed entirely.

The steady, linear growth pattern suggests that FAISS memory management handled the 10M vector load without triggering a catastrophic 2x reallocation visible at the 3-minute sampling rate.

## Key Findings

1. **Peak throughput of 914.98 QPS at concurrency 20** with 10M 768-dim vectors using HNSW (M=32, ef_search=512). QPS degrades sharply beyond concurrency 30, indicating the system is CPU-bound under heavy concurrent search load.

2. **High recall of 98.82%** (Recall@100) with NDCG of 98.99% demonstrates excellent search quality. Serial search latency averages 10ms with tight tail latencies (p99 = 15.1ms).

3. **Linear memory growth at ~3.5 GB per million vectors** with final stable RSS of 37.0 GB for 10M vectors. No memory spikes were detected at 3-minute sampling granularity, suggesting smooth memory management throughout the load.

4. **Data loading took 3h 30m** (12,610s) for 10M vectors at a sustained rate of ~793 vectors/second. Upsert throughput was consistent with ~10s per 10K-point batch initially, increasing to ~12-13s per batch as the index grew.

5. **Optimize was a no-op** (0.82s) — the server reported "No optimization needed", indicating the HNSW index was already in an optimal state for searching after the streaming inserts.
