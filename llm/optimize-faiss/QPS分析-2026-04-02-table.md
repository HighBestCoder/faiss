# HNSW 优化技术一览表

> 来源：`QPS分析-2026-04-02.md`
> 场景：IndexHNSWFlat, 768D, float32, M=32

---

| ID | 名称 | 做了什么 | 影响阶段 |
|----|------|---------|---------|
| A1 | AVX2 手写 `fvec_L2sqr_batch_4` | 把 HNSW 邻居遍历中逐个计算距离改为一次同时算 4 个邻居的 L2 距离，手写 AVX2 intrinsics | search |
| A2 | AVX-512 Batch 8/16 扩展 | 在 batch_4 基础上扩大到一次算 8 或 16 个邻居距离，利用 AVX-512 的 512-bit 寄存器 | search |
| A3 | 向量数据 Prefetch | 计算当前邻居距离时，提前 prefetch 下一批邻居的向量数据到 L2 cache | search |
| A4 | OpenMP 动态调度 | 把 HNSW 构建中的 OpenMP 调度策略从 static 改为 dynamic，减少线程间负载不均 | build |
| A5 | FP16 存储 + SIMD 距离计算 | 向量从 float32 降为 float16 存储，距离计算时用 F16C 指令加速；内存减半，cache 命中率提升 | search + 存储 |
| A6 | 空间局部性排序 | 构建后重排向量存储顺序，使图上相邻的节点在内存中也相邻 | build 后处理 |
| A7 | 激进提前终止 | 在候选堆质量足够好时提前退出搜索/构建循环，减少不必要的距离计算 | build / search |
| A8 | 邻居距离缓存 | 缓存已计算的邻居距离，避免重复计算 | build |
| A9 | 对称距离复用 | 利用 L2 距离的对称性 d(a,b)=d(b,a)，缓存并复用已计算过的距离对 | build |
| A10 | NUMA 感知内存分配 | 确保向量数据和图结构分配在查询线程所在的 NUMA 节点上，避免跨 NUMA 访问 | 部署 |
| A11 | OpenMP 条件守卫 | 当搜索 batch size=1 时跳过 OpenMP 线程池调度，避免单查询走并行区的固定开销 | search |
| B1 | VisitedTable 线程本地复用 | 把每次搜索都 malloc+memset+free 的 VisitedTable 改为 thread_local 复用，消除大块内存分配 | search |
| B2 | gRPC 线程池扩展 | 将 gRPC 服务的并发线程从 2 扩展到 16+，使多个搜索请求真正并行执行 | 部署 |
| B3 | OMP 线程池治理 | 设置 OMP_NUM_THREADS=1 + KMP_BLOCKTIME=0，避免 OMP 线程在单查询场景中自旋等待 | 部署 |
| B4 | 移除 IndexIDMap2 的 rev_map | 如果不需要 remove() 操作，用 IndexIDMap 替代 IndexIDMap2，省去 unordered_map 反向映射的内存开销 | 部署 |
| B5 | 消除 dynamic_cast | 搜索路径上有 3 次 dynamic_cast 用于设置 efSearch，改为直接写入对象字段 | search |
| C1 | Graph Reorder — BFS | 对 HNSW 图的节点按 BFS 顺序重编号，使搜索路径上相邻访问的节点在内存中连续 | build 后处理 |
| C2 | Graph Reorder — RCM | 用逆 Cuthill-McKee 算法重排节点，最小化相邻节点 ID 差距，优化 cache 行利用率 | build 后处理 |
| C3 | Graph Reorder — Weighted | 按节点被访问的频率加权重排，高频访问节点放在一起 | build 后处理 |
| C4 | HugePages (2MB 大页) | 使用 2MB 大页替代 4KB 小页，大幅减少 TLB miss | 部署 |
| C5 | THP (透明大页) | 让内核自动合并 4KB 页为 2MB 透明大页，通过 madvise 请求，无需预分配 | 部署 |
| D1 | SharedVectorStore 零拷贝重建 | 索引重建时新旧索引共享同一块向量内存，只重建图结构，不复制向量数据 | rebuild |
| D2 | DPDK 风格 4-way Batch Prefetch | 在间接寻址距离计算中，模仿 DPDK 流水线思想，同时对 4 个向量发起 prefetch | search |
| D3 | GraphReorder + Storage Compact 流水线 | 先压缩存储消除删除产生的空洞，再做图重排，形成完整的优化流水线 | build 后处理 |
